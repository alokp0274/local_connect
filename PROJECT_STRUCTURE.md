# LocalConnect Project Structure

This document explains the LocalConnect Flutter project in a practical way: what each folder contains, what each important file does, and which parts are application code versus generated Flutter scaffolding.

## 1. Project Summary

LocalConnect is a Flutter app for discovering nearby service providers such as plumbers, electricians, tutors, cleaners, and other local workers.

Current app flow:

1. App starts in `lib/main.dart`.
2. `SplashScreen` shows the branded intro.
3. User moves to `LoginScreen`.
4. User continues to `LocationScreen` and enters a PIN code.
5. `MainShell` opens the three main tabs:
   - Home
   - Add Provider
   - Profile
6. The Home tab filters local providers using in-memory dummy data.

Important implementation note:

- This is currently a frontend-focused/demo-style Flutter app.
- Authentication is dummy UI only.
- Provider data is stored in `lib/data/dummy_data.dart`.
- New providers added from the form are inserted into the in-memory list only.

## 3. Root Files Explained

### Hidden/generated root items

#### `.dart_tool/`
- Auto-generated by Flutter and Dart tooling.
- Stores package resolution, build state, and tool metadata.
- Do not edit manually.

#### `.flutter-plugins-dependencies`
- Auto-generated plugin dependency map.
- Used by Flutter tooling to connect plugins to each platform.
- Do not edit manually.

#### `.idea/`
- IDE workspace settings for IntelliJ/Android Studio.
- Mostly editor metadata, not app logic.

#### `.metadata`
- Flutter project metadata file.
- Helps Flutter tools understand project format and migration state.

### Editable root files

#### `.gitignore`
- Defines which files and folders Git should ignore.
- Usually contains build output, local IDE files, and generated artifacts.

#### `analysis_options.yaml`
- Enables `flutter_lints`.
- Controls analyzer and lint behavior.
- This project currently uses the standard Flutter lint baseline.

#### `local_connect.iml`
- IDE module definition file.
- Primarily used by JetBrains IDEs.

#### `PORT_CONFIG.md`
- Documents the fixed web development port configuration.
- Explains why port `5555` is used.
- Describes how to run the web version consistently on the same local URL.

#### `pubspec.yaml`
- Main package manifest.
- Defines:
  - project name: `local_connect`
  - version: `1.0.0+1`
  - Dart SDK constraint: `^3.11.4`
  - dependencies:
    - `flutter`
    - `cupertino_icons`
    - `google_fonts`
    - `url_launcher`
  - dev dependency:
    - `flutter_test`
    - `flutter_lints`

#### `pubspec.lock`
- Resolved package versions actually installed.
- Usually auto-updated after dependency changes.

#### `README.md`
- Still the default Flutter template readme.
- It does not yet describe the actual LocalConnect app.

#### `run_local_connect.bat`
- Windows batch launcher for the app.
- Gives three launch options:
  - Windows desktop
  - Edge web on port `5555`
  - Auto device picker with fixed web port
- Useful for quick local runs without typing Flutter commands each time.

## 4. `lib/` Application Source Code

This is the most important folder in the project. It contains the actual app logic, UI, theme, models, and demo data.

```text
lib/

|- data/

|- models/
|  |- provider_model.dart
|- screens/
|  |- add_provider_screen.dart
|  |- home_screen.dart
|  |- location_screen.dart
|  |- login_screen.dart
|  |- main_shell.dart
|  |- profile_screen.dart
|  |- provider_detail_screen.dart
|  |- provider_list_screen.dart
|  |- register_screen.dart
|  |- splash_screen.dart
|- utils/
|  |- app_theme.dart
|  |- category_icons.dart
|- widgets/
|  |- category_card.dart
|  |- custom_search_bar.dart
|  |- provider_card.dart
```

### `lib/main.dart`
- The app entry point.
- Calls `runApp(const LocalConnectApp())`.
- Sets up `MaterialApp` with:
  - title: `LocalConnect`
  - theme: `AppTheme.lightTheme`
  - home: `SplashScreen`

### `lib/data/dummy_data.dart`
- Holds hardcoded demo content.
- Defines `AppCategories` for the service categories shown in the app.
- Defines `dummyProviders`, the in-memory provider list used across screens.
- This is the current data source for provider discovery and category filtering.

### `lib/models/provider_model.dart`
- Defines the `ServiceProvider` model.
- Stores all provider fields such as:
  - id
  - name
  - service
  - phone
  - pincode
  - additional service pincodes
  - rating
  - distance
  - description
  - experience
  - image URL
- Includes helper logic:
  - `servesPincode(...)`
  - `allServicePincodes`

## 5. `lib/screens/` Detailed Explanation

### `lib/screens/splash_screen.dart`
- Animated splash screen.
- Shows branding and a visual intro.
- Uses an animation controller for fade and scale effects.
- Automatically routes to `LoginScreen` after a short delay.

### `lib/screens/login_screen.dart`
- Demo login screen.
- Uses form validation for email and password.
- Allows two actions:
  - `Log In`
  - `Continue as Guest`
- Both continue into `LocationScreen`.
- Also links to `RegisterScreen`.
- No real authentication backend is connected.

### `lib/screens/register_screen.dart`
- Demo account creation screen.
- Collects basic fields:
  - name
  - email
  - phone
  - password
- Validates the form and moves to `LocationScreen`.
- No real account is created.

### `lib/screens/location_screen.dart`
- Lets the user choose the service area using a PIN code.
- Includes quick-pick demo city/PIN options.
- After validation, opens `MainShell` and passes the selected PIN code.
- This screen is central because provider filtering depends on the chosen PIN.

### `lib/screens/main_shell.dart`
- Main tab container for the app.
- Uses an `IndexedStack` so tab state stays preserved.
- Controls three tabs:
  - `HomeScreen`
  - `AddProviderScreen`
  - `ProfileScreen`
- Receives the selected PIN code and passes it into the home experience.

### `lib/screens/home_screen.dart`
- Main discovery screen.
- Responsibilities:
  - show selected PIN code
  - show search UI
  - display service categories
  - display nearby providers
  - filter provider list by PIN code and search text
  - navigate to category list or provider detail
  - launch phone calls using `url_launcher`
- This is the core browsing experience of the application.

### `lib/screens/provider_list_screen.dart`
- Shows providers for one selected category.
- Separates results into:
  - providers in the user's PIN code
  - providers in other areas
- Uses `ProviderCard` for each list item.
- Opens `ProviderDetailScreen` when a provider is selected.

### `lib/screens/provider_detail_screen.dart`
- Full profile/details page for one provider.
- Shows:
  - provider avatar placeholder
  - service type
  - rating
  - distance
  - experience
  - description
  - contact information
  - supported service PIN codes
- Supports actions:
  - call provider
  - open WhatsApp chat link

### `lib/screens/add_provider_screen.dart`
- Form for adding a provider from the UI.
- Collects:
  - provider name
  - service type
  - phone number
  - one or more service PIN codes
  - description
- On submit, it creates a `ServiceProvider` object and inserts it into `dummyProviders`.
- This is not persistent storage; changes live only in memory while the app is running.
- Also shows a short list of recently added providers.

### `lib/screens/profile_screen.dart`
- Placeholder user profile screen.
- Displays demo user information and account-related menu items.
- Includes a logout button UI, but it has no connected logic yet.

## 6. `lib/utils/` Utility Files

### `lib/utils/app_theme.dart`
- Central theme file for colors, gradients, shadows, radii, and Material 3 styling.
- Uses `google_fonts` with the Manrope text theme.
- Defines reusable design tokens such as:
  - primary colors
  - accent colors
  - gradients
  - shadows
  - border radii
- This is the main visual system for the app.

### `lib/utils/category_icons.dart`
- Maps category names to Material icons.
- Also maps category names to color gradients.
- Used to keep category UI consistent across cards and headers.

## 7. `lib/widgets/` Reusable UI Components

### `lib/widgets/custom_search_bar.dart`
- Styled search bar used on the home screen.
- Accepts a controller and `onChanged` callback.
- Shows a clear button when text exists, otherwise shows a decorative tune/filter icon.

### `lib/widgets/category_card.dart`
- Clickable category tile.
- Used in the category grid on the home screen.
- Uses gradient colors and icons from `category_icons.dart`.

### `lib/widgets/provider_card.dart`
- Reusable summary card for a provider.
- Displays provider initials, service type, rating, distance, and quick call action.
- Used in both home and category list screens.

## 12. Architecture Notes

Current architecture style:

- Simple Flutter app without state-management packages.
- State is kept locally inside widgets using `StatefulWidget` and controllers.
- Routing is handled with `Navigator` and `MaterialPageRoute` or `PageRouteBuilder`.
- Data is in-memory and UI-driven.

Strengths of the current structure:

- Easy to understand.
- Good for UI prototyping and demos.
- Clear separation between screens, widgets, models, data, and utilities.

Current limitations:

- No backend or database.
- No persistence for added providers.
- No real auth flow.
- Very light automated test coverage.
- README does not yet reflect the actual app.

## 13. Recommended Mental Model

Use this shortcut to understand the project quickly:

- `lib/` = actual app logic
- `test/` = Flutter tests
- `pubspec.yaml` = dependencies and app metadata
- `android/`, `ios/`, `windows/`, `linux/`, `macos/`, `web/` = platform wrappers
- `build/`, `.dart_tool/` = generated output

If you want to add features, start in `lib/screens/`, `lib/widgets/`, and `lib/data/`.
